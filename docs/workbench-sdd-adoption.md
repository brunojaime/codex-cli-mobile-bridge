# Workbench SDD Adoption

This document describes how a project adopts `workbench-sdd/v1` and how LLM
actions should use the resulting artifacts.

## Ownership

Workbench owns the process contract:

- standard version resolution.
- required metadata schemas.
- artifact taxonomy.
- diagram taxonomy and protected baseline rules.
- context-pack presets and blocked-read rules.
- generated index schemas.
- doctor and Workbench health output.

The project owns the domain content:

- business language and glossary terms.
- architecture, domain, and data model content.
- feature specs, plans, tasks, diagrams, and traceability.
- module mappings and stricter context routing rules.
- additional protected baseline artifacts.

Project overrides may narrow or add context, but they must not disable
manifest-first resolution, baseline impact gates, no-broad-read behavior, or
unknown-version hard failures.

## Adoption Steps

1. Add `sdd.standard: workbench-sdd/v1` to `codex-bridge.yaml`.
2. Declare `sdd.project_type`, `sdd.domain_root`, `sdd.data_root`, and
   `sdd.generated_index_root`.
3. Declare project-owned `sdd.context_rules` with domain-to-module mappings,
   preferred context files, excluded paths, and optional stricter candidate
   limits.
4. Declare protected baseline artifacts in `sdd.protected_baseline`.
5. Run scaffold validation before writes:

```bash
python3 scripts/codex_bridge_sdd_doctor.py --workspace . --json
```

6. Use explicit write tools for remediation. The doctor is read-only by
   default.

```bash
python3 scripts/codex_bridge_sdd_backfill.py --workspace . --json
python3 scripts/codex_bridge_sdd_backfill.py --workspace . --apply --json
```

7. Confirm strict readiness:

```bash
python3 scripts/codex_bridge_sdd_doctor.py --workspace . --json --strict
```

## Required Artifacts

An adopted repo should provide:

- `codex-bridge.yaml`
- `.specify/memory/constitution.md`
- `architecture/overview.md`
- `domain/glossary.md`
- `data/persistence-model.md`
- `specs/<feature>/spec.md`
- `specs/<feature>/plan.md`
- `specs/<feature>/tasks.md`
- `specs/<feature>/traceability.yaml`
- diagram `.mmd` files plus `.yaml` metadata sidecars.
- `.sdd/spec-index.yaml`
- `.sdd/diagram-index.yaml`
- `.sdd/module-index.yaml`
- `.sdd/context-index.yaml`

Generated indexes may be regenerated by explicit backfill or context-pack flows.
They are considered stale when the manifest, specs, diagrams, traceability, or
diagram metadata change.

## Feature Workflow

New functionality should start as a feature directory under `specs/`.

1. Create `spec.md` with intent, scope, requirements, and acceptance criteria.
2. Create `plan.md` with the implementation strategy and affected modules.
3. Create `tasks.md` with reviewer-friendly implementation tasks.
4. Add feature-local diagrams under `specs/<feature>/diagrams/` when the change
   affects runtime flows, components, domain behavior, or data behavior.
5. Add `traceability.yaml` linking requirements to acceptance criteria, tasks,
   and diagrams.
6. Regenerate `.sdd` indexes before asking an LLM to implement or review.

Feature-local diagrams are the default place for new behavior. Baseline diagrams
should change only after impact is understood.

## Context Packs

Workbench LLM actions must start from a context pack. The pack includes the
manifest, standard payload, constitution, indexes, required baseline files,
selected artifacts, related specs, related diagrams, blocked reads, routing
decisions, index status, and next actions.

Required presets are:

- `new-feature`
- `modify-existing-feature`
- `bugfix`
- `architecture-change`
- `data-model-change`
- `domain-model-change`
- `implementation-from-spec`
- `diagram-update`
- `sdd-audit`

The default candidate limits are at most 5 related specs and at most 3 related
diagrams. Project `sdd.context_rules.candidate_limits` may tighten these limits
but cannot expand them beyond the Workbench default.

When indexes are missing or stale, context routing attempts deterministic
regeneration only when the caller allows it. If regeneration is unavailable or
fails, the context pack must either hard-fail or return degraded output. It must
not read every full spec body as a fallback.

## Diagram Governance

Every Mermaid diagram needs a sidecar with:

```yaml
diagram_id: components
diagram_type: components
scope: baseline
status: draft
owner: project
source: architecture/components.mmd
change_policy: baseline_impact_required
```

Baseline diagrams live under `architecture/` and require
`change_policy: baseline_impact_required`. Feature-local diagrams live under
`specs/<feature>/diagrams/` and should use impact types such as
`component-impact`, `domain-impact`, or `data-impact` when they describe a
change before baseline promotion.

Mermaid source is validated for supported top-level directives. Full render
validation is reported as skipped until a backend renderer is configured.

## Version Resolution

The canonical standard artifact is:

```text
backend/app/infrastructure/config/sdd_standards/workbench-sdd/v1.yaml
```

`SddStandardService` is the loader boundary. Backend services, doctor scripts,
Workbench API responses, and LLM instructions must use that service or a
serialized payload produced by it.

`workbench-sdd/v1` is supported. `workbench-sdd/v1.x` aliases are compatible
with v1. Unknown families or unsupported major versions are hard failures for
write flows, indexing, context packs, and Codex actions.

## SAT Pilot

SAT Catalogo Ropa is the first pilot project. It demonstrates how a Flutter app
declares its own domain language, roots, context rules, protected baselines,
diagram metadata, traceability, and generated indexes.

SAT is an example, not a template for every domain. Future projects must keep
their own glossary, modules, protected baselines, and context rules project
owned while following the Workbench process contract.

## Readiness

Final readiness means:

- doctor JSON contract returns `status: pass` in strict mode.
- no diagram metadata warnings remain.
- no missing or stale `.sdd` indexes remain.
- feature specs have traceability.
- context-pack and LLM instruction checks are ready, not blocked or degraded.
- SAT pilot strict doctor remains passing.
