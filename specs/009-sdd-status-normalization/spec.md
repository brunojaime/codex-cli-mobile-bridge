# SDD Status Normalization

## Intent

Workbench status badges must be deterministic and consistent across Codex Mobile
Bridge and every project generated by New Project Factory.

## Product Outcome

When a user opens the Workbench:

- spec cards, plan rows, and task rows show the correct state;
- plans/tasks no longer appear as `planned` or `draft` when their canonical
  tree says they are complete;
- generated projects include the same SDD tree shape as manually curated
  projects;
- future Workbench clients know exactly where to read each status from.

## Core Rule

`tree.json` is the source of truth for visual plan and task status.

`metadata.yaml` is the source of truth for spec lifecycle and aggregate task
counts.

`tasks.md` is a human-readable task index and must be generated from the same
state source as `tree.json` and `metadata.yaml`; it is not the UI source of
truth for plan/task badges.

## Scope

- Normalize current Workbench SDD artifacts that were already completed.
- Update New Project Factory so generated apps include `tree.json`, plan files,
  task files, and metadata with matching status/counts from day one.
- Keep the lazy Workbench frontend reading plan/task statuses from `tree.json`.
- Add regression coverage for generated SDD status consistency.

## Non-Goals

- Do not change the visual design of the Workbench.
- Do not introduce a new status vocabulary in this slice.
- Do not require existing generated apps to be rewritten automatically.

## Status Contract

Workbench clients read status fields as follows:

| UI element | Canonical field |
| --- | --- |
| Spec lifecycle badge | `specs/<id>/metadata.yaml.status` |
| Spec task counts | `specs/<id>/metadata.yaml.tasks` |
| Plan row badge | `specs/<id>/tree.json.plans[].status` |
| Task row badge | `specs/<id>/tree.json.plans[].tasks[].status` |

Allowed plan/task statuses are:

- `planned`
- `in_progress`
- `blocked`
- `done`

Spec lifecycle statuses may continue to use the established Workbench
vocabulary such as `draft`, `active`, `accepted`, or `validated`.

## Generated App Contract

Every new Factory project must generate:

- `specs/001-product-foundation/metadata.yaml`
- `specs/001-product-foundation/tree.json`
- `specs/001-product-foundation/plan.md`
- `specs/001-product-foundation/tasks.md`
- `specs/001-product-foundation/plans/<plan-id>/plan.md`
- `specs/001-product-foundation/tasks/<task-id>/task.md`

The files must agree on task totals and status. If a task is checked in
`tasks.md`, its corresponding `tree.json` task must be `done`.

## Acceptance Criteria

- Current lazy-loading SDD artifacts report completed work as `done`.
- New Factory projects include a SAT-like tree structure with plan/task files.
- Generated metadata counts match generated tree statuses.
- Tests fail if generated metadata and tree counts drift again.
