Codex CLI Prototype Mobile UX

Future-state mobile board for every current screen and state.

This board translates the real Flutter implementation into a polished iPhone-style design system. It covers the live chat shell, project drawer, server switching, workspace creation, voice states, attachment flows, rich assistant output patterns, and failure cases.

12 Primary screen frames based on implemented UI states
6 Distinct composer and input modes surfaced in the product
4 System overlays: drawer, workspace sheet, server sheet, source sheet
1 Unifying design language for the whole remote-control experience

Captured from the actual prototype

Screen coverage is mapped from the Flutter code in chat_screen.dart, plus the state and API layers that drive loading, pending jobs, audio, attachments, server health, workspace grouping, and unread markers.

Interaction principles

  • Codex work always feels local, live, and deeply tied to project context.
  • Heavy states such as upload, recording, and pending execution become explicit cards.
  • Project and server switching stay visible without breaking the chat mental model.

Visual direction

Midnight glass surfaces, warm activity signals, and vivid mint for trusted forward actions. The result keeps the hacker-tool energy while feeling more premium than the current utility-first shell.

Screen 12

Error, failure, and unavailable capability states

The current implementation can show chat errors, server errors, failed jobs, and voice-unavailable messaging. Treating those as a designed state keeps the product credible under real network load.

Capabilities shown
  • Inline error banners for server-level and session-level failures.
  • Failed assistant bubble using the same message model as success states.
  • Disabled voice affordance when backend audio transcription is unavailable.

UX review pass

  • Spacing is intentionally modular: all major cards use consistent 10, 12, 14, 18, 24, and 32 pixel rhythm so nothing visually collides.
  • Every screen preserves the same header and composer geometry, which makes state changes feel dependable instead of jarring.
  • Overlays are bottom sheets or side drawers only, matching the current code paths and avoiding stacked modal chaos.
  • The design improves the prototype’s clarity without inventing screens that do not exist; most “new” frames are explicit visualizations of existing states.
